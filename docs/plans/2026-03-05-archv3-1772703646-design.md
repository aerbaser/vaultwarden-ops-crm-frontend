# Vault Ops CRM Frontend Design (Task archv3-1772703646)

## 1. Objective and Delivery Envelope
Design and implement a new frontend in `/home/aiadmin/vault-crm-frontend` using Next.js 14 App Router, TypeScript, Zustand, CSS custom properties, Vitest + React Testing Library, and Playwright. The backend API is served at `http://localhost:3001` and is configured through `NEXT_PUBLIC_API_URL`.

This design intentionally targets the six explicitly required MVP routes:
1. `/login`
2. `/vault`
3. `/vault/[projectId]/[folderId]/access`
4. `/share/[token]`
5. `/operations`
6. `/audit`

The product specification contains additional domains (member lifecycle, archive/restore, favorites, versioning). Those are out of release scope for this lane but the architecture should leave clear extension points so they can be added without a structural rewrite.

## 3. Approaches Considered

### 3.1 Approach A: Page-Local Fetch + State
Each route handles its own fetches, loading states, and request errors with minimal shared infrastructure.

Pros:
- Smallest initial code footprint.
- Fast to prototype single screens.

Cons:
- High duplication of auth/session/CSRF behavior.
- Inconsistent error handling across routes.
- Difficult to enforce memory-only security guarantees consistently.
- Weak test ergonomics because domain logic is embedded in components.

### 3.2 Approach B (Recommended): Domain-Layered Client + Zustand Stores
Create a shared API client with normalized error mapping and credentials behavior, then keep route-specific domain logic in slices/hooks.

Pros:
- Centralizes security-critical behavior.
- Predictable state boundaries and easier reasoning.
- Better testing seams for store transitions and API adapters.
- Extensible to future domains without rework.

Cons:
- Slightly higher setup effort before first UI route appears complete.

### 3.3 Approach C: Heavy Server-Driven Orchestration
Rely on server components and server actions for most data workflows.

Pros:
- Can reduce client fetch boilerplate for standard CRUD.

Cons:
- Conflicts with browser-side crypto and in-memory key handling.
- Harder to model interactive decryption and dynamic vault states.
- Increased complexity for cross-route optimistic UX.

### 3.4 Decision
Use Approach B. It satisfies security constraints, testing requirements, and near-term maintainability with acceptable implementation overhead.

### 4.2 State Boundaries
- `authStore`
  - session bootstrap status
  - csrf token value and freshness
  - lock/unlock + logout actions
  - in-memory key lifecycle and secure clear
- `vaultStore`
  - projects, folders, selected node
  - encrypted sync snapshot
  - decrypted view cache keyed by item id + revision
  - create project/folder + create minimal record workflows
- `accessStore`
  - selected project/folder context
  - candidate members/groups and permission profile
  - diff preview before apply
- `shareStore`
  - share creation request state
  - resulting token metadata
  - share-view countdown and remaining views
- `operationsStore`
  - obligations, contacts, assets
  - filter/sort/grouping state
  - selected obligation and detail panel
- `auditStore`
  - paginated event list
  - filters and cursor state
- `uiStore`
  - toasts, global banners, modal visibility, transient route-level loading

### 4.3 API Client Layer
`lib/api/client.ts` defines typed wrappers over `fetch` with these invariants:
- Base URL from `NEXT_PUBLIC_API_URL`.
- `credentials: 'include'` for all authenticated flows.
- Automatic `x-csrf-token` insertion for mutating methods when token is present.
- Normalized error object with:
  - `status`
  - `code`
  - `message`
  - `recoverable`
  - `actionHint`

Any 401/403 response passes through a central auth policy hook that clears sensitive stores and coordinates redirect behavior.

### 4.4 Render Strategy
- Route pages are lightweight composition shells.
- Data-fetching and orchestration live in hooks/stores.
- Presentation components receive plain serializable props, keeping decryption and request details out of visual components.
- Suspense can be used for route-level skeletons, but critical auth and crypto steps keep explicit state-machine UI rather than hidden fallback loading.

## 5. Security and Crypto Design

### 5.1 Login Flow State Machine
`idle -> preloginLoading -> derivingKey -> tokenExchanging -> sessionBootstrapping -> success`

Error branches:
- `invalidCredentials`
- `unsupportedKdfVariant`
- `networkFailure`
- `sessionBootstrapFailure`

Each failure branch performs secure cleanup before allowing retry.

### 5.2 Sensitive Data Handling Rules
- Master password is only handled in volatile form values and immediately overwritten on completion/failure.
- Derived key material and token exchange intermediates remain in memory only.
- Never persist key material in browser storage, URLs, logs, or telemetry payloads.
- On logout, hard auth failure, or lock event:
  - clear key cache
  - clear decrypted record cache
  - clear share drafting state

### 5.3 Unsupported Variant Strategy
If backend prelogin returns KDF/provider combination not handled by the MVP path:
- stop progression before token exchange
- show deterministic error with short remediation text
- do not attempt speculative fallback derivations
- include event marker for future compatibility expansion

This keeps behavior safe and debuggable while honoring the reviewer default.

## 6. Screen-Level Design

### 6.1 `/login`
Main responsibilities:
- Collect email and master password.
- Execute crypto/session chain.
- Visualize staged progress and recoverable errors.

UI behaviors:
- Submit disabled during transition states.
- Optional "show password" toggle is purely visual and never persisted.
- On success redirect to `/vault` (or remembered intended route if already known).
- On error display action-focused message: retry, check credentials, or unsupported-profile notice.

Testing focus:
- state progression ordering
- error rendering by category
- secure reset after failure

### 6.2 `/vault`
MVP workspace with responsive multi-pane layout:
- Pane 1: projects/folders tree and create forms.
- Pane 2: synced records list with lightweight filter/search.
- Pane 3: record detail plus actions (copy/reveal/share).

Required workflows:
- project creation
- folder creation
- folder selection and sync call
- client-side decrypt for visible records
- minimal record authoring (login/card templates) to support e2e share path

Adaptivity:
- Desktop: 3-pane layout.
- Tablet/mobile: tree/list/detail stack with sticky context header and back navigation.

Failure states:
- sync failure banner with retry
- decrypt failure for individual records without crashing full list
- create flow validation and server error mapping

### 6.3 `/vault/[projectId]/[folderId]/access`
Responsibilities:
- Configure access groups/permissions for selected context.
- Preview effective changes before apply.

UX decisions:
- explicit profile selector with readable permission summary
- side-by-side "current" vs "proposed" counts for deterministic verification
- destructive-permission warning confirmation before submit

Failure handling:
- partial apply failures map to row-level warnings where possible
- full failure retains staged draft so user does not lose work

### 6.4 `/share/[token]`
Responsibilities:
- Resolve token metadata and access validity.
- Display content and usage constraints.

States:
- `loading`
- `active` (content visible)
- `expired`
- `revoked`
- `exhausted`
- `unavailable` (network/server issue)

UI must always show clear status labels and available next action (if any). TTL and view counters are visible for active state and update from backend responses.

### 6.5 `/operations`
Read-first obligations board (per default):
- fetch obligations + related assets + contacts
- group obligations by due window / status
- list and filter without requiring mutation forms
- select row to open detail panel with mapped payment/contact context

This route should include placeholder controls for future create/edit features but they are non-functional in MVP and clearly labeled.

### 6.6 `/audit`
Responsibilities:
- Render event timeline from `/api/audit/events`.
- Provide actor/type/date filtering.
- Support cursor pagination.

UX rules:
- newest first ordering
- stable row identity across incremental fetches
- refresh action preserves active filters
- expandable metadata section for affected entities and event payload summary

## 7. Data Flow and Interaction Policies

### 7.1 Fetch and Refresh
- Initial route entry performs essential fetches.
- Mutations trigger targeted revalidation of impacted resources.
- Optional background refresh (30-60s configurable) for volatile screens (`/vault`, `/audit`) can be added behind store flag.

### 7.2 Concurrency
- Use request IDs to ignore stale responses when user rapidly switches folders/filters.
- Deduplicate in-flight calls for same key parameters.
- Avoid optimistic writes where rollback is complex; prefer authoritative refetch.

### 7.3 Validation Strategy
- Synchronous schema checks before mutation requests.
- API errors mapped to field or form level based on error code.
- Unknown errors fall back to safe generic messaging plus retry.

- backend provides seeded user credentials
- API and frontend share predictable base URL config
- token TTL in test environment is long enough to complete assertions

## 11. Risks, Mitigations, and Exit Criteria

### 11.1 Key Risks
- Crypto compatibility drift between frontend assumptions and backend responses.
- Large vault sync payloads causing decryption latency on weaker client devices.
- Session + CSRF handling inconsistencies across mutations.
- Scope creep from non-MVP feature requests hidden in broader spec.

### 11.2 Mitigations
- Isolate crypto logic in adapter module with explicit unsupported branches.
- Decrypt lazily for currently viewed records, cache derived models, and defer full-list decryption.
- Enforce all request behavior through API client wrapper.
- Maintain explicit non-goals in planning docs and acceptance checks.

### 11.3 MVP Exit Criteria
The design is considered successfully implemented when:
- all six required routes are functional and adaptive
- login flow uses memory-only crypto handling and establishes session cookie
- `/vault` supports project/folder creation, sync/decrypt list, and minimal record create
- `/share/[token]` correctly renders active and terminal token states
- `/operations` and `/audit` deliver read workflows as specified
- unit suites exist for each route and pass
- Playwright login/create/share e2e passes
